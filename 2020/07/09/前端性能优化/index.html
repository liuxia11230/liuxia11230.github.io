前端性能优化

##构建优化

1.开启 gzip 压缩，gzip 压缩效率非常高。可以达到 70% 2.使用 CDN 减少打包体积

3.预渲染

##感知性能优化

1.骨架屏

2.loading

##资源优化

1.预加载
无线网后资源加载受限与网络。因此资源缓存，减少网络请求

##图片优化

1.懒加载，将真实图片 url 存在 data-url 中，只要 src 没有 url 就不会发出请求，滚动事件算处当前图片是否在可是区域。（当前元素到顶的高度-滚动条的高度<当前屏幕可是区域高度）则显示图片

2.压缩大小，减小带宽

##合理的利用缓存

1.强缓存 catch-control:max-age,public:能被浏览器和代理服务器缓存。就是给资源设置一个过时间，每次
请求都会看时间是否过期

2.强缓存失效会走协商缓存。
设置协商缓存：etag：“”，last-modified:”时间”
为什么要有 etag？
你可能会觉得使用 last-modified 已经足以让浏览器知道本地的缓存副本是否足够新，为什么还需要 etag 呢？HTTP1.1 中 etag 的出现（也就是说，etag 是新增的，为了解决之前只有 If-Modified 的缺点）主要是为了解决几个 last-modified 比较难解决的问题：

一些文件也许会周期性的更改，但是他的内容并不改变(仅仅改变的修改时间)，这个时候我们并不希望客户端认为这个文件被修改了，而重新 get；

某些文件修改非常频繁，比如在秒以下的时间内进行修改，(比方说 1s 内修改了 N 次)，if-modified-since 能检查到的粒度是秒级的，这种修改无法判断(或者说 UNIX 记录 MTIME 只能精确到秒)；

某些服务器不能精确的得到文件的最后修改时间。
